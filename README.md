# ArchLab — Learn System Design by Swapping Components

### ▶ Live: **https://aarchlab.netlify.app**

ArchLab is an interactive way to learn system design. You open a real, best-in-class
architecture, click any component to see — in plain language — the actual decision
behind it, then **swap the technology and watch what breaks downstream**. Every
explanation is hand-written and fact-checked, not generated by an LLM.

> Most learning tools are either **descriptive** ("here's what a cache is") or a
> **blank canvas** ("build your own and simulate"). ArchLab does the thing neither
> does: **counterfactual exploration on a real, opinionated design** — "here's the
> decision a system like Instagram actually made; now swap it and see the ripple
> effects, and the *why*."

## Why I built it

I was deep in system-design study (Alex Xu, *Designing Data-Intensive Applications*,
ByteByteGo) and noticed the hardest part isn't memorizing what a load balancer is —
it's understanding **why one choice is right and what breaks when you pick the other
one**. That judgment is exactly what interviews probe and what static explainers
skip.

So instead of building one more system, I built a tool that curates *many*
best-in-class designs and makes their trade-offs explorable. Teaching them well
forced me to actually understand them — the tool is both the study method and the
result. It's a portfolio piece meant to show two things at once: **system-design
judgment** (the curated, fact-checked decision content) and **product/interaction
design** (making that content genuinely fun to poke at).

## What you can do

- **Explore a complete architecture** — each domain is a full system (client, DNS,
  CDN, load balancer, API gateway, auth, services, caches, datastores, replicas,
  queues, object storage, observability), auto-laid-out left-to-right.
- **Click any component** to read what it is and why it's there — swappable or not.
- **Swap a decision** on the key nodes and watch the impact: pick a non-recommended
  option and the affected components pulse red with an "impacted" flag and red edges,
  versus a calm gold highlight for the recommended choice.
- **Quiz mode** — hides the consequences and asks you to *predict before revealing*
  (active recall, the most effective way to actually learn this).
- **Components 101** — a reference library of the ~17 building blocks, each with a
  plain ~100-word explainer and its common variants. Learn the pieces first.
- **Patterns** — the payoff: the same handful of patterns recur across every domain;
  each instance links straight to the node where it appears.
- **Clickable legend** — tap a category to learn what it is and why components group
  into it.
- Works on **desktop and phone**.

## The four domains

| Domain | The hard problem |
| --- | --- |
| Ticket & Slot Booking | Stop the same seat being sold twice when a crowd clicks at once |
| Social Media Feed | Serve a personalized feed to millions, where reads vastly outnumber writes |
| E-commerce Checkout | Never oversell inventory; never double-charge a card on retry |
| Live Betting Exchange | Push live odds, take a flood of bets at once, and settle money exactly |

## It mirrors the real interview framework

Each domain is structured like the 5-step system-design interview: scope the problem
(requirements), estimate rough numbers, design the high level (the canvas), deep-dive
the hard part (swap a decision, see the cascade), and wrap up with the one
transferable lesson the design teaches.

## Accuracy — because it's a learning source

The content *is* the product, so it's held to a high bar:

- **Two rounds of independent adversarial technical review** — one expert pass per
  domain, then a second pass verifying each fix landed and introduced no regressions.
  It caught and fixed real issues: invalid Postgres partial-index syntax, a
  fund-oversell hole in the betting flow (a stale balance read, fixed with a
  synchronous stake hold), settlement missing its results feed, DNS incorrectly drawn
  as an in-path hop, and more.
- **A plain-language pass** over every screen, defining jargon (TTL, idempotent,
  fan-out, partition, index…) the first time it appears — without dropping a single
  technical claim, caveat, or number.

Load-bearing lessons that survived the scrutiny: the partial-unique-index guarantee,
the Redlock critique, fan-out and the celebrity "hot key," idempotency (concurrent
retries + retention windows), LMAX journal-before-ack, and double-entry ledgers.

## Built with

React + TypeScript, [React Flow](https://reactflow.dev) for the canvas, and
[dagre](https://github.com/dagrejs/dagre) for automatic graph layout, bundled with
Vite. It's a fully static single-page app — **no backend, no database, no runtime
LLM calls** — deployed on Netlify, auto-redeploying on every push to `main`.
